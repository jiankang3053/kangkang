Kangkang Weather v3.0 大版本更新计划

基于网上资料、GitHub 同类项目、官方文档和本地工程只读审查整理。目标不是再打一个补丁，而是把“能用”升级成“别人电脑上也能解释、能恢复、能长期运行”的桌面自动化产品。

生成日期：2026-05-03 当前本地版本：v2.4.3 范围：计划文档，不改业务代码 重点：微信 UI 自动化、定时可靠性、天气质量、上线交付

1. 总体判断

P0最大不确定性：微信客户端 UI

这个软件的核心不是天气，而是“通过官方 Windows 微信窗口发出去”。由于没有公开发消息 API，只能走 UIAutomation/键盘/剪贴板/窗口焦点，这条路天然会受微信版本、权限级别、锁屏、弹窗、缩放、输入法、控件树变化影响。

P1当前最该升级：任务与事件台账

v2.4.3 已经能做天气、发送、记录查询历史，但“为什么没有发”仍散落在状态文件、任务计划日志、app.log、daily_task.log 里。v3 应该把所有自动化决策做成统一事件流。

P1产品形态要从工具变成控制台

现在已经有桌面 Tk、网页控制台、托盘、安装包、计划任务。下一步不是继续加按钮，而是统一成“任务控制台”：每个任务都有时间线、健康状态、失败原因、重试按钮和下一次计划。

优势基础已经够厚

已有全国地址、天气多源合并、查询记录、诊断包、微信发送测试、PyInstaller 打包、计划任务脚本、37 个自动测试。v3 可以站在这个基础上做工程化，而不是推倒重来。

一句话方向：v3.0 建议命名为 Kangkang Weather Reliability Edition，核心卖点是“可解释的微信天气自动化”：每次该不该发、有没有发、为什么失败，都能在界面里一眼看懂。

2. 外部调研结论

方向 调研发现 对本软件的启发
微信自动化生态 cluic/wxauto 明确是 Windows 微信客户端 UIAutomation 自动化，README 也声明仅用于 UIAutomation 学习交流。wxauto4 仓库页面直接写“停止更新”。另一个 pywechat 项目也强调基于 pywinauto/pure UIAutomation，并按微信版本区分支持范围。 不要押注单一第三方微信库。v3 应该做可插拔后端：当前自研 pywinauto-session 是主后端，未来可接 wxauto/pywechat 作为可选适配器，并以“微信版本指纹”决定策略。
Windows UIAutomation 边界 Microsoft UI Automation 是桌面辅助技术/自动测试框架，可以访问和操作 UI 元素；但官方说明不同用户通过 Run as 启动的进程之间不能通信。SendInput 还受 UIPI 限制，只能向相同或更低完整性级别的应用注入输入。 v3 必须把“权限同级运行”做成硬性自检；不要承诺锁屏、未登录、不同权限、完全后台可发。界面里要明确区分“环境不支持”和“程序 Bug”。
天气数据 Open-Meteo Forecast API 通过经纬度返回小时和每日预报，默认 7 天，最多 16 天；daily 聚合需要 timezone。它适合无 Key 开箱，但不能保证单源永远稳定。 当前多模型合并是正确方向。v3 可加入“数据质量分”、逐源耗时、最近预测对比、灾害/暴雨规则，以及可选中国本土天气源适配层。
托盘与桌面常驻 pystray 文档说明 Icon.run() 是阻塞调用，和其他事件循环集成要小心；Windows 默认后端支持较完整。 v3 需要单进程生命周期管理：托盘、服务、桌面窗口、网页控制台不要各自管理一套状态。建议拆成本地守护进程 + UI 客户端。
打包分发 PyInstaller 官方支持 --onefile --windowed；社区长期反馈 onefile/未签名 exe 容易触发 SmartScreen 或杀软误报。 v3 需要双轨交付：安装版优先 onedir 或 MSI/Inno，便携版保留 zip；长期要做代码签名、版本更新、校验、崩溃日志。
计划任务 Windows 计划任务支持 Daily/At 触发，也支持 StartWhenAvailable；交互式任务必须依赖已登录用户会话。本机刚发生的 0x2 就是脚本路径丢失的典型问题。 v3 应由软件自己创建、验证和修复计划任务，并把 LastRunTime、LastTaskResult、NextRunTime 直接显示出来。错过固定时间应可补偿，不应只有 70 秒窗口。

4. 潜在 Bug 与触发条件

优先级 风险/问题 触发条件 建议 v3 处理
P0 固定发送窗口过窄 _fixed_due_times 目前只接受 0-70 秒窗口。电脑睡眠、服务未运行、启动慢、天气源慢、计划任务延迟都会错过固定日报。 改为“任务账本”：计划时间、最晚补偿时间、是否已发送、失败原因分开记录。默认补偿窗口可设 30-120 分钟。
P0 微信 UI 自动化无法在锁屏/注销/不同权限下稳定运行 微信未登录、Windows 锁屏、任务跑在非交互会话、微信管理员权限而程序普通权限、或反过来。 启动前做环境门禁：会话类型、桌面是否可交互、权限级别、微信窗口 PID/完整性级别、输入法状态。失败时不尝试发送，只显示红色原因。
P0 微信版本更新导致控件标识变化 微信 3.9/4.0/4.1 的 class、automation_id、搜索结果结构不同；官方更新后控件树可能变。 新增“微信版本指纹库”：按 exe 版本、窗口 class、控件树样本选择策略；诊断包自动附 UIA 控件快照。
P1 状态 JSON 可能并发写入冲突 托盘服务、桌面端、计划任务、手动 CLI 同时运行，都可能读写 weather_poll_state.json、查询历史、缓存。 引入 SQLite 或 JSON 文件锁。推荐 SQLite：jobs、runs、sends、weather_fetches、diagnostics 统一落表。
P1 “为什么没发”仍需要翻多个文件 失败可能在计划任务、服务没启动、天气源、微信目标、输入框、发送键、免打扰、重复去重等任一层。 做“发送时间线”：计划产生、到点检查、天气获取、规则判断、微信打开、正文输入、提交验证、最终结果，一条链路显示。
P1 本地 API 缺少访问保护 服务绑定 127.0.0.1，相对安全，但本机恶意网页/插件仍可能尝试打本地接口发消息。 首次启动生成本地 token；所有写接口要求 token；Web 页面同源注入 token；诊断包不暴露 token。
P1 目标名称精确匹配与真实微信显示不一致 备注名、群名、同名好友、搜索结果包含公众号/服务号、群聊后缀、emoji、繁简体。 新增目标绑定流程：搜索候选列表展示给用户确认，保存微信目标指纹（显示名、类型、最近会话标题、可选备注）。
P1 输入框坐标聚焦仍存在误点风险 聊天输入框 click_input 失败后会尝试输入框中心坐标，遇到缩放/窗口异常可能点偏。 默认彻底关闭所有坐标点击；把坐标模式变成“高级开关 + 二次确认 + 只在测试模式可启用”。
P2 Web/Tk UI 重复实现 桌面设置、网页控制台、首次设置都各自维护交互逻辑，后续每加一个功能要改多处。 把 UI 合并到一个 Web 控制台，用 pywebview 或默认浏览器承载；Tk 只保留极简启动/错误窗口。
P2 打包 onefile 可能慢启动/误报 PyInstaller onefile 启动时解压，安全软件可能拦截；首次启动体验差。 安装版改 onedir/MSI/Inno；便携版保留 onefile 或 onedir 两种；加入代码签名和 Microsoft Defender false-positive 提交流程。
P2 天气风险规则仍偏静态 不同城市、季节、用户偏好不同；“多源分歧”可能频繁触发提醒疲劳。 给每类提醒加冷却时间、重要程度、用户反馈按钮；多源分歧只作为附加提示，不单独触发频繁补发。

5. v3 产品目标

目标 1：别人电脑上可解释

失败不再只显示“发送失败”，而是告诉用户：微信没登录、权限不一致、目标没匹配、固定时间错过、天气源超时、还是键盘提交没生效。

目标 2：自动化任务可运营

每个任务都有“计划、执行、结果、下一步”。固定日报和风险补发不再混在一起，而是分别有自己的运行策略和日志。

目标 3：跨微信版本更稳

不是写死一套控件路径，而是有版本检测、控件树采样、策略选择、失败回退、人工绑定。

目标 4：上线交付像产品

安装包、便携包、更新说明、校验、诊断包、日志、自动修复计划任务、开机自启，全套闭环。

6.2 微信自动化引擎 v3

模块设计验收标准
微信版本指纹 记录 exe 路径、文件版本、窗口 class、主窗口标题、搜索框 automation_id、输入框 automation_id、当前会话标题控件、发送按钮控件。 诊断页显示“已识别微信 3.x/4.x/未知”；未知版本进入兼容模式。
目标绑定 用户输入好友名后，软件展示搜索候选：好友/群/公众号类型、显示名、是否可输入。用户确认后保存绑定。 同名目标不会误发；找不到目标时给出候选和建议。
打开会话状态机 当前会话匹配 → 左侧会话 → 搜索框 UIA → Unicode 输入 → 候选精确匹配 → 打开后标题校验。每一步写入事件表。 失败报告能定位在具体阶段，且不会继续粘贴正文。
发送状态机 聚焦输入框 → 直接写入/剪贴板/Unicode → 内容确认 → Enter/原生 Enter/Ctrl+Enter → 输入框清空或消息出现验证。 默认不坐标点击；失败时输入框残留提示用户清理。
人工协助模式 当自动搜索失败时，提示用户手动点到聊天窗口，然后软件只做“当前会话校验 + 发送”。 复杂微信版本也能通过半自动方式稳定使用。

6.3 调度与任务系统

  • 固定日报从“瞬时触发”改成“计划运行记录”：scheduled_at、due_after、deadline_at、status、attempt_count。
  • 默认补偿策略：如果 06:00 没运行，06:00-08:00 之间启动时补发一次，并标注“补发”。
  • 风险轮询与固定日报拆表：不要把 last_check_at 同时当作轮询和日报状态。
  • 计划任务由软件创建和修复：检测脚本路径、exe 路径、工作目录、LastTaskResult、NextRunTime。
  • 增加“下次运行预测”：固定日报、轮询检查、开机自启、计划任务都在同一视图展示。

6.4 数据存储

当前 JSON 文件适合快速迭代，但 v3 多进程/计划任务/托盘/桌面并存时会有并发写入风险。建议迁移到 SQLite。
表用途
settings全局配置、版本、迁移状态。
locations地址、行政区、内部坐标、数据源。
wechat_targets目标名称、绑定状态、最后验证时间、微信版本指纹。
automation_jobs任务配置：地址、目标、时间段、固定点、提醒规则。
job_runs每次任务运行：类型、计划时间、开始/结束、状态、原因。
send_attempts每次微信发送尝试：阶段、诊断、错误分类、是否送达。
weather_fetches每次天气查询：源、耗时、失败源、缓存、质量分。
diagnostic_events统一事件流，供 UI 时间线和诊断包使用。

6.5 UI/UX 重构

  • 统一成 Web 控制台，桌面端只是启动和承载；减少 Tk 与 HTML 双维护。
  • 首页展示“今天有没有发”：最近固定日报、最近风险提醒、下一次计划、微信健康、天气源健康。
  • 任务详情页做时间线：计划生成 → 天气查询 → 规则判断 → 微信打开 → 发送 → 结果。
  • 设置页变成向导式：地址、微信目标、发送时间、提醒规则、测试发送、完成。
  • 高级设置默认可见但分组清楚：不要藏起来，也不要堆成表单迷宫。

6.6 天气与提醒升级

  • 给每次天气快照计算 quality_score：成功源数量、失败源、分歧程度、是否缓存、耗时。
  • 多源分歧不单独频繁触发，作为风险说明附在其它提醒后。
  • 新增体感温度、风力、空气质量/紫外线可选模块。
  • 消息模板支持：极简、标准、通勤、老人关怀、农业/户外。
  • 允许用户给提醒反馈“太吵/有用/漏报”，后续调整阈值。

6.7 安全与隐私

  • 本地 API 增加随机 token；写操作全部校验。
  • 诊断包默认脱敏微信好友名，只保留首字或哈希；用户可勾选“包含完整目标名”。
  • 不保存微信聊天正文，不读取无关聊天记录。
  • 安装时明确说明：需要官方 Windows 微信已登录，发送时会短暂切到前台。

  • 计划任务管理器：创建、修复、手动触发、显示 LastTaskResult。

  • 人工协助模式。
  • 完全禁用默认坐标点击，坐标模式改高级开关。
  • 真实微信矩阵测试：微信 3.9、4.0、4.1；Win10/Win11；100%/125%/150% 缩放。
v3.0-rc上线交付
  • Web 控制台重构，桌面端只承载/启动。
  • 安装包改良：开始菜单、卸载、计划任务注册、开机自启管理。

  • 可选通知渠道：桌面通知、企业微信/钉钉/Webhook，仅作为备用，不替代个人微信。

8. 测试与上线门槛

测试类型v3 必须覆盖通过标准
单元测试 调度补偿、SQLite 迁移、天气质量分、提醒冷却、目标绑定、错误分类。 核心模块覆盖率 80%+，关键状态机分支全覆盖。
UI 自动化 mock 测试 微信窗口消失、搜索框缺失、候选不匹配、权限不一致、输入框不可读、发送后未清空。 每类失败都有明确错误分类和用户建议。

计划任务测试 准点、错过后补发、用户未登录、电脑睡眠、手动触发、脚本路径丢失。 界面能解释所有失败；任务修复按钮可恢复。
打包测试 便携版、安装版、无源码路径运行、卸载、升级旧配置、杀软扫描。 干净 Windows 账号首次运行可完成设置；zip 内不缺脚本和 README。
安全测试 本地 API token、诊断包脱敏、配置导出、非同源网页调用。 外部网页不能直接触发发微信；诊断包不泄露完整聊天内容。
上线红线：只要“固定日报没发但界面无法解释原因”还存在，就不应称为上线版。v3 的第一目标不是更多功能，而是把每次自动化行为变成可追踪、可恢复、可复盘。
